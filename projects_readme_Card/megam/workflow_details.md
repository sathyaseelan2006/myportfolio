# Megam Workflow & System Architecture Details

This document outlines the workflow and architectural processes currently implemented in the **Megam Air Quality Monitor** project.

---

## 1. Core Workflow Sequences

### A. Location Initialization & Coordinate Resolution
1. **User Input:** The user types a city name in the search bar or clicks a coordinate point directly on the WebGL 3D globe (`react-globe.gl`).
2. **Reverse Geocoding:** The search request is processed by `geocodingService.ts` utilizing the **OpenStreetMap Nominatim API** to fetch exact latitude, longitude, city name, and country name.
3. **Globe Redirection:** The camera position on the 3D globe shifts dynamically using Three.js controls to center on the coordinates.

---

### B. Hybrid Data Fusion (The Waterfall Engine)
When coordinates are resolved, `satelliteService.ts` executes a structured, priority-based data fetch sequence to guarantee global coverage:

```
[Coordinate Resolved]
         │
         ├──► 1. Query IQAir API (Primary Commercial Data)
         │       └── Success? Return AQI + Weather.
         │
         └──► 2. Query OpenAQ API (via Vercel Serverless Proxy)
                 └── Success? Return official ground station pollutants.
                 │
                 └──► 3. Fallback: Query NASA MODIS (Aerosol Optical Depth)
                         └── Success? Calculate AQI from satellite estimates.
                         │
                         └──► 4. Last Resort: Query WAQI API
                                 └── Fallback: Return nearest data cluster.
```

*Note: If data is retrieved from a station different from the exact clicked coordinates, the system calculates the distance (Haversine formula) and displays a confidence value.*

---

### C. Client-Side Neural Network Time-Series Prediction
When the user triggers the "Forecast" panel, the application runs the ML flow:

1. **Check Cache:** The system searches **IndexedDB** for an existing pre-trained model for the specific location key (`indexeddb://aqi-model-{locationKey}`).
2. **Dynamic Training Data Collection:**
   * If no model is found, the application fetches **180 days of history** from `dataCollectionService.ts`.
   * It aggregates ground measurements from **OpenAQ** and fills remaining holes using hourly CAMS history models from **Open-Meteo**.
3. **Background Training:**
   * It initializes **TensorFlow.js** and starts a background LSTM (Long Short-Term Memory) training sequence.
   * Features normalized: **AQI, PM2.5, PM10, O3, NO2, SO2, CO**.
   * Sequence length window: 7 days.
4. **Save Weight Binaries:** The model weights are saved to **IndexedDB** for future instantaneous inference.
5. **Heuristic Fallback:** If the client browser is on low-spec hardware or historical records are sparse, it falls back to a statistical seasonal moving average via `simplePredictionService.ts`.

---

## 2. Inactive & Archival Feature Integrations

To ensure 100% honesty, the following modules are present in the codebase but are **not currently active or used** in the main user experience:

*   **Supabase Relational Client (`lib/supabaseClient.ts`):** Contains connections and TS typings for a comments and reviews postgres schema (`reviews`). This was completely deleted from active React components on **December 25, 2025** to allow for a zero-login, locally-stored user experience.
*   **Google Gemini Service (`services/geminiService.ts`):** Exposes `getAqiDataForLocation` and `getAqiDataForCoordinates` to query the `gemini-2.5-flash` model for narrative health advisories. While environment variables are configured in Vite, these service functions are **not imported or executed** in the main `App.tsx` layout.

---

## 3. Storage Scheme

*   **LocalStorage:** Stores search query history, favorite tracking coordinates, and analytical consent.
*   **IndexedDB:** Caches compiled neural networks (`aqi-model-*`) generated by TensorFlow.js.
